# Practice Board

## What Exists Right Now

The hub already has public `S / E / T` structure, but starter tasks are the first active layer, not the final full curriculum.

This page explains:

- what a practical task is
- how to interact with it
- when to fork
- when to open a pull request

---

## How To Use Practice In KMH

There are three common ways to interact with a task:

### `1. Solo Practice`

Use this when you want to:

- test your interest
- build understanding
- create notes or a mini artifact

Typical outputs:

- short summary
- cleaned README section
- mini dashboard
- lightweight checklist

### `2. Fork Practice`

Use this when you want to:

- try the task in your own space first
- experiment without pressure
- build a draft before contributing back

Typical outputs:

- forked repo or branch
- prototype
- notebook
- draft markdown

### `3. Contribution Practice`

Use this when you are ready to improve the shared surface.

Typical outputs:

- pull request
- docs improvement
- issue cleanup
- visualization
- reproducibility note

## Starter Task Types

### `Read & Map`

- summarize a lane
- make a glossary
- build a resource map

### `Think & Decide`

- compare options
- frame a venture problem
- decide on the best starting point

### `Build Something Small`

- create a checklist
- make a tiny dashboard
- prepare a simple workflow

### `Improve Shared Assets`

- edit docs
- improve readability
- add examples
- fix structure

---

## When To Fork

Fork first if:

- you are unsure
- the task is exploratory
- you want to test your own version
- you might break structure while learning

---

## When To Open A PR

Open a PR when:

- your output improves shared clarity
- your change is readable by others
- the task is marked `safe-for-fork` or `academy-practice`
- you can explain what changed and why

---

## Good Practice Rule

Start with:

`one direction -> one small task -> one visible output`

That is enough to create momentum.
